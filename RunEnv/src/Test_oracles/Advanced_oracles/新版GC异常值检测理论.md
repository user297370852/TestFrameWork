## Part1：三个通道解决相对排名指标固有的问题

​	回顾相对排名指标，使用它来检测异常值，存在一系列无法避免的问题，下面针对这些问题逐一构建解决方案：

### 问题一：弱势GC长期排名过低，无法检测排名靠后的GC的异常

我们引入一个更能反映相对于**历史基准表现**退化的指标——regret：
$$
\text{regret}(c,m,g)=\log\left(\frac{x(c,m,g)}{x_{best}(c,m)}\right)
$$
其中，c表示测试用例case，m表示指标metric，g表示gc类型，x就是具体的值，其中x-best也就是当前测试用例，所有GC类型下该指标的最优值，注意，**regret是按JDK版本进行区分的**；

由于我们的测试数据已经剔除了0值（例如GC次数为0），并且极端大的值正是我们想要检测出来的，因此不需要平滑常数（regret的极端值不会影响到历史基线的构建，参考下文，我们都是采用中位数构建）

然后我们基于历史基线定义Z-self,这是统计学上常用的离群值判断指标：
$$
Z_{self}=\frac{\text{regret}-\text{median-regret}}{1.4826\cdot \text{mad-regret}+\epsilon}
$$

- median_regret：历史上（同一 GC、同一指标）所有 regret 的中位数。
- mad_regret：绝对中位差，一种鲁棒的波动估计；**它是由历史样本中每个样本减去中位数，取绝对值，排序，取中位数得到，表示历史上典型的绝对偏差**，它与当前regret无关
- 乘 1.4826 是为了让 MAD 在正态分布下等价于标准差，直接对应正态分布下的Z-score。这样做可以使分母近似历史数据的标准差，可以直接使用经典Z-score的阈值，例如阈值3对应99.7%,即使数据不完全正态，该计算方式仍有参考性（regret的特性，一般情况下正态分布的拟合效果都比较好）

Tips:中位数和 MAD 对异常值不敏感，而 GC 数据中可能存在罕见的大退化（正是我们要检测的），如果使用均值和标准差，这些罕见值会严重拉高均值和标准差，导致后续检测阈值失效。因此我们选择稳健统计量。

- ϵ：极小值，防除零和极大结果，这是因为mad_regret可能很小，若Z-self极大会直接覆盖其余两个通道，同时会使最终score非常大，对“最终score越大，越可能是异常“这一假设是不利的

我们取Z-self的退化方向来判断异常，例如使用阈值3，此时检测变为：
$$
Z_{self}^+=\max(0,Z_{self}) >3
$$
下面用简单的例子展示它的计算方式：

假设我们有两个 JDK 版本：JDK 11 和 JDK 17。
运行同一个测试用例 `c`。
关注的指标 `m` = `gc_stw_time_ms`（GC 总暂停时间，单位 ms，越小越好）。
每个 JDK 下测试 3 种 GC：G1、Shenandoah、ZGC。

**步骤 1：获得原始数据**

| JDK 版本 | GC 类型    | gc_stw_time_ms |
| :------- | :--------- | :------------- |
| JDK 11   | G1         | 120            |
| JDK 11   | Shenandoah | 150            |
| JDK 11   | ZGC        | 80             |
| JDK 17   | G1         | 100            |
| JDK 17   | Shenandoah | 80             |
| JDK 17   | ZGC        | 70             |

**步骤 2：分别计算每个 JDK 下的 `x_best`**

- 对于 **JDK 11**：min(120, 150, 80) = **80**（ZGC 最优）
- 对于 **JDK 17**：min(100, 90, 70) = **70**（ZGC 最优）

//若不同JDK混合计算：x-best很小，早期JDK的regret普遍偏大

**步骤 3：计算每个 GC 的 regret**

- **JDK 11 下**：
  - G1: log⁡(120/80)=log⁡(1.5)≈0.405log(120/80)=log(1.5)≈0.405
  - Shenandoah: log⁡(150/80)=log⁡(1.875)≈0.629log(150/80)=log(1.875)≈0.629
  - ZGC: log⁡(80/80)=0log(80/80)=0
- **JDK 17 下**：
  - G1: log⁡(100/70)=log⁡(1.4286)≈0.357log(100/70)=log(1.4286)≈0.357
  - Shenandoah: log⁡(90/70)=log⁡(1.2857)≈0.251log(90/70)=log(1.2857)≈0.251
  - ZGC: log⁡(70/70)=0log(70/70)=0

**注意**：JDK 11 和 JDK 17 下的 regret 是独立计算的，即使数值接近，也是分别基于各自的 `x_best`。你不能说 JDK 11 的 G1 (regret=0.405) 比 JDK 17 的 G1 (regret=0.357) 差，因为它们的 baseline 不同（JDK 11 的最优是 80，JDK 17 的最优是 70），跨 JDK 比较 regret 没有意义。这正是设计意图：**每个 JDK 有自己的行为模式，不跨 JDK 比较**。（这是基于测试经验的一种设计，跨JDK版本的回归异常大多数情况下是默认参数修改、底层算法更新导致的，我认为强行让regret也能解决跨版本回归的异常检测容易适得其反）

//不同JDK、GC不能混合计算的理由，理论（例如底层算法改进），实验。

**步骤 4：历史基线如何建立**

对于 `(jdk_version=11, metric=gc_stw_time_ms, gc_type=G1)`，我们收集历史上多次运行（不同测试用例或不同轮次）的 regret 值，假设有 100 个样本：

```
regret_samples = [0.40, 0.42, 0.38, 0.45, 0.41, ...]  # 单位：自然对数
```

然后计算：

- `median_regret` = 这些样本的中位数，比如 0.41
- `mad_regret` = 绝对中位差，比如 0.03
- `q99_regret` = 99% 分位数，比如 0.55

这些就是该 `(jdk_version, metric, gc_type)` 的基线。

当新来一个测试用例（同样是 JDK 11，指标 gc_stw_time_ms，GC 类型 G1），我们算出当前 regret = 0.60，则：
$$
Z_{self}= \frac{0.60−0.41}{1.4826×0.03}≈4.27
$$
超过 3，触发异常。

//**极小值的选择，或者，有没有办法去掉极小值？**

上述设计就是**通道A：自基线退化信号**，现在看它是如何解决问题一的：

通道A不看排名，看 **regret**（当前GC比同用例最优GC慢多少倍）。
对于SerialGC，即使它长期垫底，它的regret仍然有一个**自身的历史分布**（比如历史上regret在2.0~3.0之间波动）。
如果这次它的regret跳变到5.0，那么 Zself就会很大，触发告警。

此外，**如果所有GC都明显退化，这种方式是否也能判断异常？**答案是肯定的：

历史正常考试（测试用例1）：

| GC类型     | 暂停时间(ms) |
| :--------- | :----------- |
| ZGC (最优) | 100          |
| G1         | 150          |
| SerialGC   | 800          |

计算regret：

- ZGC regret = 0
- SerialGC regret = log(800/100) = log(8) = 2.08

当前“难题”考试（测试用例2，内存压力大）：

所有GC暂停时间都增加了，但比例可能不同。

| GC类型     | 暂停时间(ms)      |
| :--------- | :---------------- |
| ZGC (最优) | 600 (变差6倍)     |
| G1         | 900（变差6倍）    |
| SerialGC   | 1800 (变差2.25倍) |

计算regret：

- ZGC regret = 0（它自己是最优）
- SerialGC regret = log(1800/600) = log(3) = 1.10

**关键观察**：

- 历史上SerialGC的regret ≈ 2.08，当前regret = 1.10，**反而变小了**（因为SerialGC恶化幅度小于最优GC的恶化幅度，相对差距缩小了）。
- 通道A计算 Zself是一个**负数**，取 Zself+=0，**不会告警**。

**结论**

通道A **不会** 把“所有GC都变差但差生相对差距缩小”的情况误报为异常。
它只关注**当前regret是否显著大于自身历史regret**，即**差生相对于最优的退化程度是否比平时更严重**。
因此：如果变差比例都相同，regret没有变化，Z-self很小甚至为0，不会记为异常；如果所有GC都变差，但是某个GC变差特别严重，此时的regret才会变大，导致Z-self变大，被检测为异常



### 问题二：指标的绝对差距很小，但是排名变化很大

**其实regret已经解决这个问题了，指标的绝对差距很小，regret一定很小**

但如果我们仍然要保留rank这一排名信息，也可以通过通道B解决，但在此之前先讨论一下保留rank的意义：

首先从统计验证角度看，通道A是regret+Z-self判据，regret是连续型变量，而通道B使用的是离散型变量rank，并且是基于统计直方图频数分布判断的，**两者是正交统计量，可以相互验证，提高置信度**

此外，相对排名可以捕捉“竞争力下降”这一信息，例如“某个测试用例下其他GC都进步了，而某GC没有进步甚至退步，导致排名变化大”，这种情况有可能不会被regret检出，但rank可以指示出**一个GC不适用某些场景**这种信息。

因此设计了基于rank的通道B（当然，我们可以把通道B的权重设置的比较小）：

首先定义历史尾部概率：
$$
p_{rank}=P(R\ge r_{obs}|m,v,g)
$$

- R：随机变量，表示该 `(metric, jdk_version, gc_type)` 在历史运行中出现的排名（1 表示最优，数值越大越差）。
- r-obs：当前观测到的排名（例如当前 GC 在本次测试中，在该指标下的排名）。
- **P(R≥r-obs)**：从历史数据中计算排名 **大于等于** r-obs 的比例。

为了计算这个概率，我们只需要维护一个直方图，记录该（m,v,g）条件下的历史频数，例如：

| 排名 (rank) | 出现次数 | 比例 |
| :---------- | :------- | :--- |
| 1           | 120      | 0.12 |
| 2           | 340      | 0.34 |
| 3           | 500      | 0.50 |
| 4           | 40       | 0.04 |
| 5           | 0        | 0.00 |
| **总计**    | 1000     | 1.00 |

$$
例如，r_{obs}=3,那么p_{rank}=P(R\ge3)=0.54,因此p_{rank}越小，代表历史上越罕见，排名越极端
$$

然后做一个对数变换方便处理（同时，你可以注意到，**取对数后这个S的值与通道A的Z-self+的值大小更接近了**，并且都有越大越能说明离群的特征）：
$$
S_{rank}^{raw}=-log(p_{rank}+\epsilon)
$$
这里的p-rank可能为0，且为0是有实际意义的——这种情况在历史上从未出现，因此需要一个极小值epsilon做平滑，当p-rank为0时这个S会非常大。

这已经可以用来检测相对排名指标的离群值了，接下来我们定义幅度门控，用来解决问题2:
$$
\text{gate}=\min\left(1, \frac{\text{regret}}{\tau_m}\right)
$$
这里的regret如前文所述，可以衡量当前值与组内最优值的差异，而τ_m则是一个门控阈值，它的意思是我们觉得多大的 regret 才算有实际退化”：

例如对于 `gc_stw_time_ms`，τ=0.10（意味着 regret ≥ 0.10 即慢约 1.1 倍以上才认为有意义），此时若regret很小，gate会变得很小，而取min是限制gate的最大值，最终通道B的得分为：
$$
S_{rank}=S_{rank}^{raw}\cdot gate
$$
这样，实际差异不大时，排名异常会被gate压制下去，从而解决了问题2

### 问题三：极端值不够突出

相对排名指标还存在极端值不够突出的问题（因为我们始终希望，当检测算法给出一个异常得分后，得分越高的越异常，越可能指向真实存在的BUG），因此我们建立**通道C：极端值通道**，通道C的目的是：**在同一个测试用例内，检测某个 GC 的指标是否远差于其他 GC**。它不依赖历史数据，只关心当前用例内不同 GC 之间的横向比较，专门捕捉绝对值上的极端异常。

同样的我们对原始数据取对数：
$$
y(c,m,g)=log(x(c,m,g))
$$
(取对数：前两个通道都是取对数计算，保持原始数据处理方式统一(**是否所有指标都适合取对数？**；乘法差异变加法差异；极端值本身绝对差值大偏态严重，取对数做压缩)

然后这里我们计算一个**横向Z分数**：
$$
Z_{tail}=\frac{y(c,m,g)-\text{median}_g(y)}{1.4826\cdot MAD_g(y)+\epsilon}
$$
这个式子与通道A的Z-self很类似，但是由历史纵向比较改为例同一测试用例中的横向比较：

- **median_g(y)：在同一个测试用例 c 中，**所有 GC 类型的 y 值的中位数。
- **MADg(y)**：这些 y值的绝对中位差，计算方式与通道 A 中相同，但这里是在 **同一用例内横向** 计算。
- **ϵ**：极小正数（如 1e-9），防止除零（例如几种GC的GC次数完全一样）。
- **1.4826**：同前，将 MAD 缩放为近似的标准差。

当然，我们只需要取退化方向：
$$
Z_{tail}^+=max(0,Z_{tail})
$$
举例计算：

| GC 类型    | x (ms) | y = ln(x) |
| :--------- | :----- | :-------- |
| ZGC        | 100    | 4.6052    |
| Shenandoah | 120    | 4.7875    |
| G1         | 130    | 4.8675    |
| Serial     | 500    | 6.2146    |

- *y* 值列表：4.6052, 4.7875, 4.8675, 6.2146
  排序：4.6052, 4.7875, 4.8675, 6.2146
  中位数 = (4.7875 + 4.8675)/2 = **4.8275**
- 每个值与中位数的绝对偏差：
  - |4.6052-4.8275| = 0.2223
  - |4.7875-4.8275| = 0.0400
  - |4.8675-4.8275| = 0.0400
  - |6.2146-4.8275| = 1.3871
- 排序绝对偏差：0.0400, 0.0400, 0.2223, 1.3871
  MAD = 中位数 = (0.0400 + 0.2223)/2 = **0.13115**

然后计算Z-tail，显然分母为1.4826×0.13115≈0.1944，从而：

- ZGC: (4.6052 - 4.8275)/0.1944 = (-0.2223)/0.1944 = -1.143 → Z-tail+=0
- Shenandoah: (4.7875 - 4.8275)/0.1944 = (-0.04)/0.1944 = -0.206 →Z-tail+= 0
- G1: (4.8675 - 4.8275)/0.1944 = 0.04/0.1944 = 0.206 → Z+tail+=0.206
- Serial: (6.2146 - 4.8275)/0.1944 = 1.3871/0.1944 = **7.135** → Z-tail+=7.135

当然最后我们还可以附加一个实际限制条件，例如：
$$
x(c,m,g) > \lambda_m \cdot x_{mid}(c,m)
$$
也就是说我们可以附加设置原始值必须大于组内中位数的lambda倍时Z-tail才有效（否则直接输出0），因为MAD可能很小，这种情况下绝对差异不大也可以使Z-tail较大，附加限制条件避免了这种“统计上离群，实际上差异不大”的情况；同时该条件可以针对不同GC分别配置，弥补各GC本身的差异



### 总结

三个通道都会输出一个分值，最终我们可以加权求和作为所谓“异常值”,例如：
$$
S=0.60Z_{self}^++0.15S_{rank}+0.15Z_{tail}^+
$$
上述的原理阐述是基于**同版本JDK、不同GC类型**计算的，实际上我们可以迁移以实现跨版本回归检测：
$$
regret(c,m,g)\to regret(c,m,v)
$$
把GC类型参数换为JDK版本参数即可得到**同一GC在不同JDK版本下的**历史基线，并进行异常检测
