# 基于统计排名基准模型的GC异常检测测试预言设计方案

## 一、研究背景与动机

### 1.1 现有测试预言的问题

通过分析现有的三个测试预言（`performance_anomaly.py`、`stw_anomaly.py`、`gc_count_anomaly.py`），我们发现其核心问题在于**阈值的设置缺乏统计学依据**，主要表现为：

| 预言类型 | 现有阈值设置方式 | 问题分析 |
|---------|----------------|---------|
| 性能异常 | 固定倍数阈值（如5倍、10倍、15倍） | 忽略了GC类型间的固有性能差异；不同JDK版本下GC性能特征不同 |
| STW异常 | 固定绝对阈值（如ZGC: 2ms, G1GC: 100ms） | 未考虑测试程序的GC压力差异；阈值缺乏统计学验证 |
| GC次数异常 | 固定倍数阈值（如6倍、8倍中位数） | 未建立正确的GC次数分布模型；未区分正常波动与异常 |

### 1.2 核心洞察

基于我们对10,506个测试文件的统计分析，得出以下关键洞察：

1. **排名分布具有可预测性**：同一GC在同一JDK版本下的排名近似服从正态分布
2. **排名期望(μ)和标准差(σ)可从统计中获得**：这为异常检测提供了统计学基准
3. **异常的定义应基于偏离基准的程度**：而非任意的固定阈值

## 二、理论基础

### 2.1 排名分布模型

#### 2.1.1 排名的定义

对于单个测试用例，在JDK版本 \(v\) 下，共有 \(n_v\) 种GC参与测试。按指标值排序后，GC \(g\) 获得排名 \(r_g \in \{1, 2, ..., n_v\}\)。

#### 2.1.2 排名分布假设

**假设1（排名正态性假设）**：对于大量独立测试用例，GC \(g\) 在JDK版本 \(v\) 下的排名近似服从截断正态分布：

\[
R_{g,v} \sim \mathcal{TN}(\mu_{g,v}, \sigma_{g,v}^2, 1, n_v)
\]

其中：
- \(\mu_{g,v}\)：GC \(g\) 在JDK版本 \(v\) 下的平均排名（从统计数据获得）
- \(\sigma_{g,v}\)：排名的标准差（从统计数据获得）
- \(n_v\)：JDK版本 \(v\) 下支持的GC数量
- 截断范围：\([1, n_v]\)

#### 2.1.3 验证分析

从统计数据观察排名标准差：

| JDK版本 | GC类型 | 执行时间排名μ | 执行时间排名σ | GC暂停时间排名μ | GC暂停时间排名σ |
|--------|--------|-------------|-------------|----------------|----------------|
| 21 | SerialGC | 1.952 | 1.452 | 4.700 | 1.495 |
| 21 | ZGC | 4.703 | 1.081 | 1.378 | 0.834 |
| 21 | ShenandoahGC | 4.995 | 1.216 | 1.750 | 0.836 |

**关键发现**：
- 排名标准差 \(\sigma\) 通常在 0.6 ~ 1.7 之间
- 低延迟GC（ZGC、ShenandoahGC）在延迟指标上标准差更小（表现更稳定）
- 吞吐量GC（SerialGC、ParallelGC）在执行时间上标准差较大（受程序特性影响更大）

### 2.2 异常检测模型

#### 2.2.1 Z-Score异常检测

对于单次测试中GC \(g\) 获得的排名 \(r\)，计算Z-Score：

\[
Z_{g,v} = \frac{r - \mu_{g,v}}{\sigma_{g,v}}
\]

**异常判定**：
- \(|Z_{g,v}| > k\) 时判定为异常
- 通常取 \(k = 2\)（95%置信度）或 \(k = 3\)（99.7%置信度）

#### 2.2.2 排名偏移方向的意义

| 偏移方向 | 含义 | 示例 |
|---------|------|------|
| \(Z > +k\) | 排名显著靠后（表现比预期差） | ZGC执行时间排名应为4-5，但实际排名为1-2（比预期好） |
| \(Z < -k\) | 排名显著靠前（表现比预期好） | SerialGC执行时间排名应为2左右，但实际排名为5-6（比预期差） |

#### 2.2.3 双侧异常检测的意义

- **正向异常**（排名比预期好）：可能揭示编译器优化、JIT热点等
- **负向异常**（排名比预期差）：更可能是真正的bug或性能回归

### 3.3 各预言的详细设计

---

### 3.3.1 执行时间排名异常检测预言

#### 设计目标

检测执行时间排名显著偏离基准的GC测试。

#### 检测算法

```
输入: test_results, baseline_model
输出: anomaly_report

算法:
1. 按JDK版本分组测试结果
2. 对于每个JDK版本v:
   a. 计算各GC的执行时间排名r_g
   b. 对于每个GC g:
      i.   从baseline获取 (μ_{g,v}, σ_{g,v})
      ii.  计算Z-Score: Z = (r_g - μ_{g,v}) / σ_{g,v}
      iii. 若 |Z| > k (k=2或3)，标记为异常
3. 聚合所有异常，生成报告
```

#### 异常分类

| 类型 | Z-Score范围 | 含义 | 可能原因 |
|------|------------|------|---------|
| 严重欠表现 | Z > 3 | 排名比预期差3个标准差以上 | JIT编译失败、内存泄漏、线程竞争 |
| 中度欠表现 | 2 < Z ≤ 3 | 排名比预期差2-3个标准差 | 编译优化不足、GC参数问题 |
| 正常波动 | -2 ≤ Z ≤ 2 | 排名在预期范围内 | 无异常 |
| 中度超表现 | -3 ≤ Z < -2 | 排名比预期好2-3个标准差 | 特殊优化生效、测试用例特性 |
| 显著超表现 | Z < -3 | 排名比预期好3个标准差以上 | 可能存在JIT优化或测试问题 |

#### 与原预言对比

| 方面 | 原预言 | 新预言 |
|------|--------|--------|
| 阈值依据 | 固定倍数(5,10,15倍) | 统计分布的Z-Score |
| GC区分 | 简单分类 | 每个GC独立基准模型 |
| JDK版本 | 不区分 | 按版本独立建模 |
| 可解释性 | 低（为什么是15倍？） | 高（99.7%置信度） |

### 3.3.2 GC暂停时间排名异常检测预言

#### 设计目标

检测GC暂停时间排名显著偏离基准的测试。

#### 检测算法

```
输入: test_results, baseline_model
输出: anomaly_report

算法:
1. 过滤无GC活动的测试（gc_stw_time_ms == 0）
2. 按JDK版本分组
3. 对于每个JDK版本v:
   a. 计算各GC的STW时间排名r_g
   b. 对于每个GC g:
      i.   从baseline获取 (μ_{g,v}, σ_{g,v})
      ii.  计算Z-Score
      iii. 检测异常
4. 额外检测：
   a. 低延迟GC一致性检测：ZGC与ShenandoahGC排名应接近
   b. GC类型交叉验证：排名顺序是否符合设计预期
```

#### 特殊规则：GC类型关系验证

基于统计分析，我们建立了GC类型的预期排名关系：

**延迟指标（GC暂停时间、最大暂停时间）**：
\[
\text{E}[R_{EpsilonGC}] < \text{E}[R_{ZGC}] < \text{E}[R_{ShenandoahGC}] < \text{E}[R_{ParallelGC}] < \text{E}[R_{G1GC}] < \text{E}[R_{SerialGC}]
\]

**执行时间指标**：
\[
\text{E}[R_{SerialGC}] < \text{E}[R_{EpsilonGC}] \approx \text{E}[R_{ParallelGC}] \approx \text{E}[R_{G1GC}] < \text{E}[R_{ZGC}] < \text{E}[R_{ShenandoahGC}]
\]

#### 与原预言对比

| 方面 | 原预言 | 新预言 |
|------|--------|--------|
| 绝对阈值 | 固定值(ZGC:2ms, G1:100ms) | 排名相对基准 |
| GC对比 | G1不应比Serial长20倍 | Z-Score检测偏离 |
| 跨版本对比 | 相邻版本比较 | 每版本独立基准 |
| 特殊处理 | 无 | 过滤无GC活动用例 |

---

### 3.3.3 GC次数排名异常检测预言

#### 设计目标

检测GC次数排名显著偏离基准的测试。

#### 检测算法

```
输入: test_results, baseline_model
输出: anomaly_report

算法:
1. 过滤条件：total_gc_count > 5（排除GC活动过少的用例）
2. 过滤无GC活动用例（所有GC count == 0）
3. 按JDK版本分组
4. 计算各GC的GC次数排名
5. 使用Z-Score检测异常
```

#### 统计基准

从分析结果看，GC次数排名的分布特征：

| JDK版本 | GC类型 | 平均排名μ | 标准差σ | 预期排名范围(μ±2σ) |
|--------|--------|----------|--------|-------------------|
| 21 | EpsilonGC | 1.000 | 0.000 | [1, 1] (恒定第一) |
| 21 | ShenandoahGC | 1.499 | 0.766 | [0, 3] |
| 21 | ZGC | 1.555 | 0.865 | [0, 3] |
| 21 | SerialGC | 4.838 | 1.429 | [2, 6] |

#### 与原预言对比

| 方面 | 原预言 | 新预言 |
|------|--------|--------|
| 阈值设置 | 中位数的6-8倍 | 排名Z-Score |
| 稳定性假设 | Serial最稳定 | 从数据中学习 |
| 分组方式 | 按GC类型 | 按JDK版本×GC类型 |
| 异常定义 | 绝对值超阈值 | 相对排名偏离 |

## 四、置信度与阈值选择

### 4.1 Z-Score阈值选择

| 阈值k | 置信度 | 双侧覆盖率 | 适用场景 |
|------|--------|-----------|---------|
| 1.5 | 86.6% | 13.4%异常率 | 宽松检测，用于探索性分析 |
| 2.0 | 95.4% | 4.6%异常率 | **推荐**：平衡检测率与误报率 |
| 2.5 | 98.8% | 1.2%异常率 | 保守检测，减少误报 |
| 3.0 | 99.7% | 0.3%异常率 | 严格检测，仅报告极端异常 |

### 4.2 多指标联合检测

当同一测试用例在多个指标上同时出现异常时，应提高其优先级：

\[
\text{Priority} = \sum_{m \in M} w_m \cdot |Z_m|
\]

其中 \(M\) 是出现异常的指标集合，\(w_m\) 是指标权重：

| 指标 | 权重 \(w_m\) | 理由 |
|------|------------|------|
| gc_stw_time_ms | 1.0 | 核心延迟指标 |
| max_stw_time_ms | 0.9 | 重要延迟指标 |
| duration_ms | 0.7 | 综合性能指标 |
| total_gc_count | 0.5 | 辅助诊断指标 |

### 4.3 分层检测策略

```
Level 1 (宽松): k = 2.0, 目的是发现潜在问题
    ↓ 若检测到异常
Level 2 (严格): k = 3.0, 确认异常严重程度
    ↓ 若仍为异常
Level 3 (交叉验证): 检查其他指标是否也异常
```

## 七、风险评估与缓解

### 7.1 潜在风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 排名正态性假设不成立 | 检测准确性下降 | 可使用核密度估计(KDE)替代正态分布 |
| 测试用例样本不足 | 基准不稳定 | 设置最小样本量阈值(如n>100) |
| 新JDK版本无基准 | 无法检测 | 使用最近版本的基准作为参考 |
| 特殊测试用例干扰 | 基准偏移 | 使用鲁棒统计量(中位数、MAD) |

### 7.2 后续研究方向

1. **非参数方法**：使用核密度估计(KDE)或分位数回归，避免正态性假设
2. **在线学习**：基准模型随新数据动态更新
3. **多变量联合检测**：考虑指标间的相关性
4. **因果分析**：异常定位到具体代码变更

---

## 八、结论

本设计方案提出了一种基于**统计排名基准模型**的GC异常检测方法，核心创新在于：

1. **从固定阈值转向统计基准**：使用Z-Score替代经验阈值
2. **建立版本×GC的细粒度模型**：每个(JDK版本, GC类型)组合独立建模
3. **统一的异常检测框架**：三种预言共享相同的数学基础
4. **可解释的检测结果**：Z-Score提供明确的统计置信度

该方法将GC差分测试的异常检测从"经验驱动"提升为"数据驱动"，具有更高的准确性、可解释性和可维护性。
