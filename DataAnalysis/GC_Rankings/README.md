# GC运行时间排名分析

## 概述

本分析基于Java GC差分测试结果，统计不同GC类型在不同JDK版本下各指标的排名。分析数据来自 `results/20260101` 目录下的测试日志。

### 分析方法

1. 对每个测试用例的每次成功运行，按JDK版本分组
2. 在同一JDK版本内，按指定指标值排序计算各GC的排名
3. 排名规则：指标值越小排名越靠前（排名值越小），相同值给予相同排名
4. 统计各GC的平均排名、测试次数和标准差

### 支持的分析指标

| 指标 | 字段名 | 描述 | 单位 | 过滤无GC用例 |
|------|--------|------|------|:------------:|
| 执行时间 | `duration_ms` | 程序总执行时间 | ms | 否 |
| GC暂停时间 | `gc_stw_time_ms` | GC总暂停时间(STW) | ms | 是 |
| 最大单次暂停时间 | `max_stw_time_ms` | 最大单次GC暂停时间 | ms | 是 |
| GC次数 | `total_gc_count` | GC总次数 | 次 | 是 |

> **重要说明**: 对于GC相关指标（暂停时间、GC次数），我们**排除了所有GC指标值都为0的测试用例**。这是因为许多简单测试程序根本没有触发GC，此时所有GC的指标都是0，无法进行有意义的排名比较。

## 关键发现

### 1. 执行时间（吞吐量）

- **SerialGC** 在大多数JDK版本中排名第一，这是因为其实现简单、开销低
- **EpsilonGC** 紧随其后，因为它是一个不做实际GC操作的实验性收集器
- **JDK 17是个特例**：G1GC排名第一（平均排名1.648），这可能是因为G1在JDK 17中经过深度优化
- **ZGC** 和 **ShenandoahGC** 排名靠后，因为低延迟GC有更高的运行时开销

### 2. GC暂停时间（低延迟）

- **EpsilonGC** 始终排名第一（平均排名1.000），因为它不做实际GC
- **ZGC** 和 **ShenandoahGC** 表现优异，这是低延迟GC的设计目标
- **SerialGC**、**ParallelGC**、**G1GC** 暂停时间较长，排名在3-5位之间

### 3. 最大单次暂停时间

- 排名趋势与GC暂停时间类似
- **ZGC** 的最大暂停时间控制最为稳定（标准差较小）
- 低延迟GC在设计上就追求最大暂停时间的可预测性

### 4. GC次数

- **EpsilonGC** 永远为0次（不触发GC）
- **ShenandoahGC** 和 **ZGC** GC次数较少
- **SerialGC** GC次数最多，因为需要频繁进行压缩

### 5. 数据质量说明

- 约有**30%**的测试用例实际触发了GC活动
- 对于GC相关指标的分析，我们排除了无GC活动的用例，确保排名的有效性
- 执行时间指标使用全部测试数据

## 注意事项

1. 本分析基于**多个指标的单独排名**，不代表GC的整体性能
2. 测试用例主要为短时间运行的Java程序，可能无法反映生产环境的GC表现
3. 低延迟GC（ZGC、Shenandoah）的优势在大内存、高并发场景下才能体现
4. EpsilonGC是实验性GC，仅适用于无内存分配压力的场景
5. **排名越低表示在该指标上表现越好**
6. **GC相关指标已过滤无GC活动的测试用例**，确保排名的有效性
